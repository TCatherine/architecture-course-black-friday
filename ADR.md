### Схема коллекций MongoDB и выбор шард-ключей (Задание 7)

Принцип выбора ключей для shard:
- кардинальность (много уникальных значений)
- равномерность записи (нет hotspot-шарда)
- изолированность запросов (в запросе присутствует ключ для shard'a, что позволяет идти сразу на нужный шард)
- отсутсвие монотонного роста (отсутствие ситуации, когда значения шард-ключа всегда увеличиваются по порядку)

#### Схема коллекции `orders`

```json
{
  "_id":          "ObjectId",
  "customer_id":  "ObjectId",
  "created_at":   "ISODate",
  "items": [
    {
      "product_id": "ObjectId",
      "name":       "String",
      "price":      "Decimal128",
      "quantity":   "Number"
    }
  ],
  "status":       "String",
  "total_amount": "Decimal128",
  "geo_zone":     "String"
}
```

Основные операции:
- Быстрое создание заказов с одновременным списанием остатков.
- Поиск истории заказов конкретного пользователя.
- Отображение статуса заказа.

**Шард-ключ:** `customer_id с хешированием`

Причины:
- поддерживает высокую кординальность
- обеспечивает равномерное распределение записей по шардам
- по customer_id мы можем вычислить хеш и найти шарду (так как в основных операциях опеределен customer_id)
- хеширование позволяет избежать монотонного роста

#### Схема коллекции `products`

```json
{
  "_id":      "ObjectId",
  "name":     "String",
  "category": "String",
  "price":    "Decimal128",
  "stock": [
    { "geo_zone": "moscow",       "quantity": 150 },
    { "geo_zone": "ekaterinburg", "quantity": 50  },
    { "geo_zone": "kaliningrad",  "quantity": 30  }
  ],
  "attributes": {
    "color": "String",
    "size":  "String",
    "brand": "String"
  }
}
```

Поле `stock` представлено массивом объектов, а не вложенным документом — это позволяет атомарно обновлять остаток конкретной геозоны через `arrayFilters`:

```js
db.products.updateOne(
  { _id: productId },
  { $inc: { "stock.$[zone].quantity": -1 } },
  { arrayFilters: [{ "zone.geo_zone": "moscow" }] }
)
```

Основные операции:
- Частые обновления остатков при покупках.
- Поиск товаров по категориям и фильтрация по диапазону цен.
- Описание товара на странице продукта.

**Шард-ключ:** `category, _id:`

Причины:
- Поиск по категории считается основным запросов -> в ключе шарда должен быть `category`, для быстрого поиска шарда
- Основная операция - обновление остатков при покупках -> в ключе шарда должен быть `_id`, для быстрого поиска шарда и для высокой кординальности внутри каждой категории

#### Схема коллекции `carts`

```json
{
  "_id":        "ObjectId",
  "owner_key":  "String",
  "user_id":    "ObjectId | null",
  "session_id": "String | null",
  "items": [
    {
      "product_id": "ObjectId",
      "quantity":   "Number"
    }
  ],
  "status":     "String",
  "created_at": "ISODate",
  "updated_at": "ISODate",
  "expires_at": "ISODate"
}
```

Коллекция содержит как гостевые корзины (есть `session_id`, нет `user_id`), так и пользовательские (есть `user_id`, нет `session_id`). Введено синтетическое поле `owner_key`, которое всегда заполнено: для гостей — значением `session_id`, для авторизованных пользователей — строковым представлением `user_id`.

Основные операции:
- Cоздание корзины, когда заходит гость или новый пользователь.
- Получение текущей корзины по фильтру { session_id, status:"active" } или { user_id, status:"active" }.
- Добавление или замена товара в корзине.
- Удаление товара из корзины.
- Слияние гостевой корзины в пользовательскую, если пользователь залогинится:
- Отметка корзины как заказанной.

**Шард-ключ:** `owner_key с хешированием`

Причины:
- поддерживает высокую кординальность
- обеспечивает равномерное распределение записей по шардам
- по customer_id мы можем вычислить хеш и найти шарду (так как в основных операциях опеределен customer_id)
- хеширование позволяет избежать монотонного роста

| Сущность                | Требование к консистентности | Частота записи | Нужно ли переносить в Cassandra | Обоснование                                                                                                                  |

#### Концептуальные модели (Задание 10.2)

##### Модель корзины

```sql
CREATE TABLE carts (
    owner_key    TEXT,
    cart_id      UUID,
    status       TEXT,
    user_id      UUID,
    session_id   TEXT,
    items        LIST,
    created_at   TIMESTAMP,
    updated_at   TIMESTAMP,
    PRIMARY KEY (owner_key, cart_id)
) WITH default_time_to_live = 604800;
```

Partition key `owner_key` по аналогии с MongoDB. Cтрока с высокой кардинальностью (UUID или случайный session ID). Хеш Cassandra равномерно раскидывает партиции по узлам.

Clustering key `cart_id` — у одного пользователя несколько корзин в разных статусах. Clustering key позволяет хранить их в одной партиции и фильтровать по `status` в приложении.
##### Модель истории заказов

```sql
CREATE TABLE orders_by_customer (
    customer_id  UUID,
    month_bucket TEXT,
    created_at   TIMESTAMP,
    order_id     UUID,
    status       TEXT,
    total_amount DECIMAL,
    geo_zone     TEXT,
    PRIMARY KEY ((customer_id, month_bucket), created_at, order_id)
) WITH CLUSTERING ORDER BY (created_at DESC, order_id ASC);
```

Partition key  `(customer_id, month_bucket)` позволяет ограничивать размер партиции.
Clustering key `created_at DESC, order_id` — сортировка по убыванию даты, чтобы последние заказы читались первыми без сортировки в приложении

##### Модель пользовательских сессий

```sql
CREATE TABLE user_sessions (
    session_id   UUID,
    user_id      UUID,
    owner_key    TEXT,
    data         TEXT,
    created_at   TIMESTAMP,
    updated_at   TIMESTAMP,
    PRIMARY KEY (session_id)
) WITH default_time_to_live = 86400;
```

Partition key `session_id` — UUID гарантирует максимальную кардинальность и равномерное распределение. При добавлении нового узла Cassandra переносит только часть виртуальных узлов, а не все данные.

Clustering key не нужен — одна сессия, один документ.

#### Стратегии восстановления целостности (Задание 10.3)

Основные стратегии восстановления целостности:
- Hinted Handoff
- Read Repair
- Anti-Entropy Repair

##### Стратегии восстановления целостности для корзины

Стратегия: Hinted Handoff + Read Repair.

Hinted Handoff был выбран, так как критична операция добавления товара в корзину. Если реплика временно недоступна, корзина не должна «потерять» товар.
Read Repair был выбран, так как каждое открытие страницы корзины — это чтение, которое автоматически согласует реплики.

Консистентность записи: локальный кворум — подтверждение от большинства реплик. Даёт гарантию, что сессия не потеряется при падении одного узла.

##### Стратегии восстановления истории заказов

Стратегия: Hinted Handoff + Read Repair + Anti-Entropy Repair.

Hinted Handoff — обязательно, так как запись заказа должна дойти до всех реплик, даже если одна из них временно недоступна.
Read Repair — полезен, потому что пользователи часто проверяют историю заказов. Каждое чтение органически восстанавливает консистентность реплик.
Anti-Entropy Repair — обязателен по расписанию, для гарантии полноты истории

Консистентность записи: полный кворум — данные во всех репликах.

##### Стратегии пользовательских сессий

Стратегия: Hinted Handoff.

Сессии живут 24 часа и после истечения TTL автоматически удаляются. Если узел упал, Hinted Handoff доставит запись, когда он вернётся — этого достаточно. Read Repair и Anti-Entropy Repair для сессий избыточен: тратить ресурсы на согласование данных, которые через несколько часов удалятся, нецелесообразно.

Консистентность записи: локальный кворум — подтверждение от большинства реплик. Даёт гарантию, что сессия не потеряется при падении одного узла.
